# Cross-chain bridging — research + engine choice (2026-06-01)

Goal: move value from one chain to another (e.g. Polygon→Arbitrum/Solana, BTC↔EVM). Distinct from same-chain
swap (already built). Needs a bridge.

## Engine chosen: LI.FI
One API spanning EVM + Solana + native Bitcoin (routes BTC through Thorchain internally), 60+ chains, 18+ bridges,
tunable integrator fee. Keyless for quotes. Sources: li.fi/, docs.li.fi, quicknode bridge-aggregator roundup 2026.
Alternatives noted: Rango (widest non-EVM, BTC→EVM single-sig), Squid (100+ chains), Thorchain (native BTC), deBridge (fast EVM↔Solana). LI.FI picked for one-API breadth + fee hook + our existing EVM sign+broadcast fits.

## Confirmed live (2026-06-01)
GET https://li.quest/v1/quote?fromChain=&toChain=&fromToken=&toToken=&fromAmount=&fromAddress=&integrator=chain-signer
→ returns `transactionRequest` {to,data,value,chainId} + `estimate.toAmount`. Tested: 1 USDC Polygon→Arbitrum via
Across, toAmount 0.991195 USDC, tx.to = LI.FI diamond 0x1231DEB6f5749EF6cE6943a275A1D3E7486F4EaE. The tx is a
standard EVM tx → sign with our EVM wallet + broadcast via the existing recipe (REUSE: sign-and-broadcast pattern).

## GOTCHA (critical)
LI.FI sits behind Cloudflare. The default Python urllib user-agent is BLOCKED → HTTP 403 "error code: 1010".
FIX: send a browser User-Agent header (Mozilla/5.0 … Chrome …). Caught via /stuck (don't conclude "blocked" on a 403).
Same Cloudflare lesson as the Polymarket curl_cffi note.

## Build plan (phase 2)
- bridge.py: get_bridge_quote(...) [LI.FI /quote + UA header + integrator + fee]; bridge_evm(wallet, quote, nonce, fees, broadcast) [sign the returned transactionRequest with our EVM wallet + broadcast]. Reuses sign-and-broadcast + injectable-network patterns from CATALOG.md.
- Integrator fee: LI.FI `integrator` string + `fee` param (share, e.g. 0.001 = 0.1%). Confirm exact param at build.
